# Задача: миграция на SQLite

## Проблема
Текущее хранилище — JSON-файл (`notifications.json`) — имеет несколько ограничений:

1. **Нет индексов**: поиск по `datetime` (метод `getAllForNow`) требует полного сканирования массива.
2. **Проблемы с параллельным доступом**: одновременные запросы `saveOne` могут привести к конфликту чтения/записи (race condition).
3. **Блокировка файла**: каждая операция записи перезаписывает весь файл.
4. **Масштабируемость**: при большом количестве уведомлений файл растёт, операции замедляются.
5. **Отсутствие транзакций**: если `sendMany` или `removeMany` завершатся с ошибкой, данные могут остаться в непредсказуемом состоянии.

## Желаемое поведение
Заменить JSON-хранилище на встраиваемую СУБД SQLite, которая обеспечит:

- **Индексы** для быстрого поиска по дате.
- **Атомарные операции** и транзакции.
- **Конкурентный доступ** (readers‑writer locks на уровне SQLite).
- **Один файл базы** (как и сейчас), простота бэкапа и переноса.
- **Поддержку JSON‑полей** (храним `payload` и `subscription` как TEXT с JSON).

Новое хранилище должно реализовывать тот же интерфейс `NotificationStore`, чтобы остальные модули (`scheduler`, `router`) не потребовали изменений.

## Риски
1. **Зависимость на нативную библиотеку**: `better‑sqlite3` требует компиляции под целевую платформу (может быть проблемой в Docker).
2. **Миграция данных**: существующие уведомления из `notifications.json` нужно перенести в SQLite, иначе они потеряются.
3. **Производительность**: SQLite может быть медленнее на очень частых операциях записи (но для планировщика это не критично).
4. **Тестирование**: потребуются новые unit‑тесты, возможно, с временной in‑memory базой.
5. **Сложность отладки**: данные теперь лежат в бинарном файле, для просмотра нужен SQLite‑клиент (либо консольная утилита, либо GUI).

## Принятые решения
- Использовать `better‑sqlite3` (синхронный, но с поддержкой промисов) — быстрее `sqlite3` и активно поддерживается.
- Хранить JSON‑полей как TEXT, использовать `JSON.stringify` / `JSON.parse` при чтении/записи.
- Создать индекс `idx_notifications_datetime` для ускорения `getAllForNow`.
- Не добавлять миграцию данных автоматически — при первом запуске с SQLite база будет пустой, а старый `notifications.json` останется нетронутым (можно позже написать скрипт миграции при необходимости).
