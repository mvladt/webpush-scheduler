# План: миграция на SQLite

## Шаги реализации

### 1. Установка зависимости
```bash
npm install better-sqlite3
```
Добавить в `package.json` в `dependencies` (или `devDependencies`? — для работы приложения нужна, поэтому `dependencies`).

### 2. Создание модуля SQLite-хранилища
**Файл:** `src/sqliteStore/store.ts`

Реализовать фабрику `createSqliteStore(filename?: string)`, возвращающую объект, удовлетворяющий `NotificationStore`.

**Структура модуля:**
- Импорт `better‑sqlite3`, создание экземпляра `Database`.
- При первом запуске — создать таблицу `notifications` и индекс.
- Методы `saveOne`, `removeOne`, `removeMany`, `getOneById`, `getAllForNow`.
- Валидацию `datetime` (можно вынести в общую утилиту, чтобы не дублировать код JSON‑хранилища).
- Закрытие соединения при остановке приложения (опционально, можно оставить открытым до конца работы процесса).

**Пример схемы:**
```sql
CREATE TABLE IF NOT EXISTS notifications (
  id TEXT PRIMARY KEY,
  payload TEXT NOT NULL,
  datetime TEXT NOT NULL,
  subscription TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_notifications_datetime ON notifications(datetime);
```

**Примечания:**
- Поля `payload` и `subscription` хранить как JSON‑строки (`JSON.stringify`).
- Для `getAllForNow` потребуется переписать логику фильтрации «сейчас ±2 минуты» в SQL‑запрос.

### 3. Обновление точки входа
**Файл:** `src/main.ts`

Заменить:
```ts
const store = createJsonStore("notifications.json");
```
на
```ts
const store = createSqliteStore("notifications.db");
```

**Альтернатива:** оставить возможность выбора хранилища через переменную окружения (например, `STORE_TYPE=sqlite|json`), но для простоты можно сразу перейти на SQLite.

### 4. Написание тестов
**Файл:** `tests/sqliteStore/store.test.ts`

Скопировать структуру тестов из `tests/jsonStore/store.test.ts`, но:
- Использовать временную in‑memory базу (`:memory:`) для изоляции тестов.
- После каждого теста закрывать соединение и удалять файл (если использовался файл).

**Дополнительные тесты:**
- Проверка создания индекса.
- Проверка конкурентных запросов (если нужно).
- Проверка корректности SQL‑запроса `getAllForNow`.

### 5. Обновление типов (если потребуется)
**Файл:** `src/types.ts`

Сейчас `NotificationStore` — интерфейс, который уже подходит. Никаких изменений не нужно.

### 6. Создание скрипта миграции (опционально)
**Файл:** `scripts/migrate-json-to-sqlite.ts`

Утилита, которая:
1. Читает `notifications.json`.
2. Подключается к `notifications.db`.
3. Вставляет все записи в таблицу `notifications`.
4. Сохраняет старый файл как `notifications.json.backup`.

Запуск вручную, если нужно перенести исторические данные.

### 7. Обновление документации
**Файл:** `README.md`

Добавить раздел «Хранилище» с описанием:
- Используется SQLite, файл `notifications.db`.
- Как просмотреть данные (например, через `sqlite3 notifications.db`).
- При первом запуске создаётся пустая база, старые JSON‑данные не переносятся автоматически.

### 8. Проверка работы
- Запустить `npm run dev`, убедиться, что сервер стартует.
- Отправить тестовое уведомление через API.
- Проверить, что уведомление появилось в базе (через SQLite‑клиент).
- Убедиться, что планировщик корректно находит и отправляет уведомления.

## Файлы для изменения
1. `package.json` — добавить `better-sqlite3` в `dependencies`.
2. `src/sqliteStore/` — новый директорий с `store.ts` и, возможно, `tools.ts` (общие валидации).
3. `src/main.ts` — заменить `createJsonStore` на `createSqliteStore`.
4. `tests/sqliteStore/` — новый директорий с тестами.
5. `README.md` — обновить описание хранилища.
6. (опционально) `scripts/migrate-json-to-sqlite.ts` — скрипт миграции.

## Порядок выполнения
1. Установить зависимость.
2. Написать `src/sqliteStore/store.ts` (без тестов).
3. Подменить хранилище в `main.ts` и проверить вручную.
4. Написать тесты.
5. Обновить документацию.
6. (По желанию) создать скрипт миграции.

## Оценка времени
- Разработка хранилища: 2–3 часа.
- Написание тестов: 1–2 часа.
- Интеграция и проверка: 1 час.
- Документация и скрипт миграции: 1 час.

**Итого:** около 5–7 часов.
