# Задача: настроить CI/CD через GitHub Actions

## Проблема

Сейчас в проекте нет автоматизации:

1. **Нет CI.** Тесты (`env`, `jsonStore`, `integration`, `playwright`) запускаются только вручную локально. Если в push/PR попадает регресс — он обнаруживается только при следующем ручном прогоне.
2. **Нет автоматического деплоя.** Чтобы обновить `scheduler.push.mvladt.ru`, нужно зайти на сервер и вручную выполнить `git pull` / `docker compose up -d --build`. Это медленно, легко забыть шаг, нет аудита того _что_ и _когда_ задеплоили.
3. **Нет защиты ветки `main`.** В неё можно зафорс-пушнуть, можно влить непротестированный код.
4. **Нет автообновления зависимостей.** `package.json` и actions со временем устаревают, копятся CVE.
5. **Нет аудита безопасности.** Уязвимости в npm-пакетах и в собственном коде не отслеживаются.

## Желаемое поведение

После push в `main` (или merge PR в `main`):

1. Запускаются все тесты в чистом окружении (Ubuntu + Node 22).
2. Если тесты прошли — собирается Docker-образ и публикуется в **GHCR** (`ghcr.io/mvladt/webpush-scheduler:<tag>`) с тегами `latest`, коммит-SHA и semver (при наличии git-тега).
3. Образ автоматически выкатывается на production-сервер (`scheduler.push.mvladt.ru`).
4. После выката проверяется healthcheck — если приложение не отвечает, статус деплоя падает.

После push в feature-ветку или открытия PR:

- Прогоняются только тесты + typecheck. Docker не собирается, деплой не выполняется.

## Принятые решения

### Хостинг runner'ов — GitHub-hosted

Свой self-hosted не нужен: проект маленький, бесплатных минут хватит с запасом. Если репозиторий публичный — минуты вообще не тратятся.


### Стратегия деплоя — SSH + systemd (нативно, без Docker)

> **Обновлено после проверки сервера.** На SPB (188.225.37.62) Docker не установлен. Текущий scheduler работает как голый `node --env-file .env src/main.ts` без systemd, рядом стоит `mvladt-nuxt` под systemd. Для согласованности с уже сложившейся инфраструктурой выбираем **нативный деплой**:

1. На сервере — каталог `/root/projects/webpush-scheduler/` (там и сейчас лежит).
2. Создаётся systemd-юнит `webpush-scheduler.service` по образцу `mvladt-nuxt.service`.
3. CI заходит по SSH под `root` (как уже настроено) и делает:
   - `git fetch && git reset --hard origin/main`
   - `npm ci --omit=dev`
   - `systemctl restart webpush-scheduler`
   - `curl --fail https://scheduler.push.mvladt.ru/api/health` (внешний smoke-check)

Альтернативы и почему отвергнуты:
- **Docker + GHCR.** Требует поставить Docker на VPS (3.9 ГБ RAM — потянет, но overhead не нужен ради одного контейнера). Расходится с подходом соседнего `mvladt-nuxt`. **Оставляем как вариант на будущее** — Dockerfile в репо сохраняется, но в CD не используется.
- **rsync вместо git.** Меньше зависимостей на сервере, но git уже там, и `git reset --hard` атомарнее.
- **Watchtower / GitOps** — нерелевантно без Docker.
- **Self-hosted runner на VPS** — лишняя нагрузка, изоляция, не нужно.

### Реестр образов — GHCR (отложено)

Раз деплой нативный, публикация Docker-образа в GHCR не обязательна. Оставляем в плане как **опциональный этап** для будущего (например, если когда-нибудь поднимем Docker).

### Тестирование Playwright в CI — отдельная стратегия

Текущий `playwright.config.ts` использует **headed Chrome через X11** (`headless: false`, `--ozone-platform=x11`). В CI X11 нет. Варианты:

- **A.** Включить `headless: true` через `CI=1` env-переменную и установить браузеры через `npx playwright install --with-deps chromium`.
- **B.** Использовать официальный Docker-образ `mcr.microsoft.com/playwright`.
- **C.** Не запускать Playwright в CI вообще — оставить только локальный smoke.

Решение: **A** — самое чистое, унифицирует локальный и CI-запуск. Для локалки оставим `headless: false` (удобно отлаживать руками), для CI переопределим через env.

### TypeScript-проверка — `tsc --noEmit`

`tsconfig.json` уже настроен с `noEmit: true`. В CI достаточно добавить `tsc` без аргументов как отдельный job — он провалится, если есть тип-ошибки. Это бесплатная линтер-замена при отсутствии ESLint.

### Линтер/форматтер — отложено в отдельную задачу

ESLint + Prettier добавим позже отдельной задачей (см. `eslint-prettier.task.md`). В первой итерации CI ограничиваемся `tsc` как статической проверкой.

### Обновления зависимостей — Dependabot

Встроен в GitHub. Конфиг `.github/dependabot.yml` для npm + GitHub Actions. Раз в неделю PR'ы с обновлениями — CI их сам прогонит.

### Безопасность — несколько слоёв

1. **Pin-by-SHA для actions** (Dependabot будет обновлять). Защита от supply chain.
2. **Минимальные `permissions`** в workflow (по умолчанию `contents: read`, расширяем по необходимости).
3. **GitHub Environments** для prod с required reviewer — опционально, если хочется ручное подтверждение.
4. **Branch protection на `main`**: запрет force-push, обязательные status checks.
5. **CodeQL** — бесплатное SAST-сканирование (для публичных репо).
6. **`npm audit`** в CI на high+ уязвимости.

### Версионирование релизов — теги

Простой подход: руками тегаем коммиты `vX.Y.Z`, Docker-образ получает соответствующий тег. Никаких semantic-release/release-please — это можно прикрутить позже, если ритм релизов вырастет.

## Риски

1. **Утечка SSH-ключа деплоя.** Митigation: отдельный ed25519-ключ только для CI, ограниченный через `command="...",no-port-forwarding` в `authorized_keys`. Хранится в GitHub Secrets.
2. **Деплой ломает прод.** Митigation: после `systemctl restart` — внешний health-check; если не отвечает — workflow падает красным; откат через `git reset --hard <предыдущий-SHA>` руками или отдельным workflow.
3. **Гонка между текущим nohup-процессом и systemd-юнитом.** При первом включении systemd надо явно убить старый процесс. Делается одноразово.
4. **Утечка `.env`.** Файл уже на сервере, не в git, не в CI. SSH-ключ в Secrets шифруется GitHub'ом.
5. **Playwright flaky в CI.** Зависит от реального Push Service (Google/Mozilla). Митigation: пробуем включить, если поплывут — выносим в опциональный job по label.
6. **Аутентификация под `root` через SSH из CI.** Это не лучшая практика (лучше отдельный deploy-user), но соответствует тому, как сейчас устроены другие сервисы на SPB. Митigation на потом — завести `deploy`-пользователя с правами только на нужные операции.
7. **GitHub Actions минуты** — лимита нет для публичного репо.
